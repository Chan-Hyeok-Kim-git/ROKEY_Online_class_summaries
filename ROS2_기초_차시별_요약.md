# 01장: 컴퓨터 구조, 네트워크와 통신
로봇의 어원: 로보타(체코어): 천한 노동, 중노동, 강제 노동 등 등 뜻함

### 로봇의 역사
최초의 로봇: 유니메이트(금속 사출물 운반) → 로봇 발전:(소프트뱅크 페퍼, 아이보, 휴보 등) → 미래:키오스크, 휴머노이드 등
로봇 3원칙:아이작 아시모프의 런어라운드

### 컴퓨터 구조
컴퓨터 부팅: 전원 인가 → POST(Power-On Self-Test) 실행 → 바이오스 설정 로드 → 부트 로더 실행 → 운영체제 커널 로드 → 초기 프로세스 실행 → 사용자 인터페이스 실행

바이오스 단계: ROM에 저장되어 ROM-BIOS라 부름; 기본적 하드웨어(키보드, 디스크)등 상태 확인 후 부팅 장치 선택해 부팅 디스크의 첫 섹터의 512B 로드함. 이 512B가 마스터 부트 레코드(MBR)이라 하고, 여기는 디스크 어느 파티션에 2차 부트 로더가 있는지에 대한 정보가 저장되어 있어 부트로더 찾아 메모리에 로드하는 작업까지 수행.

CPU란: 중앙처리장치(Central Processing Unit), 산술 논리 연산 유닛(Arithmetic Logic Unit), Control Unit, Register

폰 노이만 구조: 프로그램 내장 방식의 컴퓨터. CPU, 메모리, 프로그램 세 가지 요소로 구성, CPU 메모리 분리, 이 둘을 연결하는 버스.

프로그램: 어떤 작업을 하기 위해 해야할 일들 순서대로 나열한 것. 보통 보조 기억장치(HDD, SSD)에 저장<br>
프로세스: 프로그램 실행돼 돌아가고 있는 상태, 동적인 상태의 프로그램. 코드, 데이터, 스택, 힙의 구조로 이루어진 독립된 메모리 영역 할당 받음. 최소 하나의 스레드 가짐<br>
스레드: 프로세스가 할당 받은 자원 이용하는 실행 단위, 프로세스 특정 수행 경로 및 실행되는 여러 흐름 단위, 독자적인 스택 메모리 가짐(그러나 코드, 데이터, 힙 메모리 공유)

메모리 영역: 코드 영역, 데이터 영역, 스택 영역, 힙 영역

스택
- 매우 빠른 액세스
- 변수를 명시적으로 할당 해제 할 필요가 없다
- 공간은 CPU에 의해 효율적으로 관리된다
- 지역 변수
- 스택 크기 제한 (OS에 따라 다름)
힙
- 변수를 전역적으로 액세스 할 수 있다
- 메모리 크기 제한 없음
- (상대적으로) 느린 액세스
- 효율적인 공간 사용을 보장하지 못하면 메모리 블록이 할당 된 후 시간이 지남에 따라 메모리가 조각화되어 해제 될 수 있다
- 메모리를 관리해야 함(변수를 할당하고 해제하는 책임있음)

API(Application programming Interfae): 응용 프로그램에서 운영 체제나 프로그래밍 언어가 제공하는 기능 제어할 수 있게 만든 인터페이스<br>
라이브러리: 코드 묶음. 독립성 가짐. 프로그램이 능동적으로 사용(필요할 때 호출). <br>
프레임워크: 프로그램 솔루션 개발 수월하게 하기 위해 제공된 소프트웨어 환경, 기반 시설 담당해주는 것이 프레임워크 역할(예: 파이토치, 유니티, 텐서플로, Vue, 장고, 스프링 등)<br>
라이브러리 vs 프레임워크: 프레임워크가 주도권을 쥐고 개발자가 프레임워크의 규칙에 따라 코딩해야함. 라이브러리는 개발자가 마음대로 호출 가능.

API vs 라이브러리: API는 컴포넌트 사용하는 규악 및 호출 위한 수단이라 구현 로직 필요없음, 라이브러리가 컴포넌트 그 자체라 구현로직이 존재

인터프리터 컴파일러: 고급언어 vs 저급언어<br>
인터프리터는 한 줄 씩 해석해서 실행, 컴파일러는 전체 소스코드 한 번에 기계어로 번역해 실행

XML(eXtensible Markup Language): 공유 가능한 방식으로 데이터 정의하고 저장(자체적으로 컴퓨팅 작업 수행 불가)<br>
ROS에서 XML 역할: ROS 패키지 빌드 및 의존성 관리해줌

모뎀(MOdulator and DEModulator):신호 변조해 송신하고 수신측에서 원래의 신호로 복구하기 위해 복조하는 장치<br>
TCP:응용 프로그램이 데이터 교환할 수 있는 네트워크 대화 설정하고 유지한 방법 정의하는 표준<br>
UDP:IP를 사용하는 네트워크 내에서 컴퓨터 간 메세지 교환 때 제한된 서비스만을 제공한 통신 프로토콜. TCP의 대안으로 쓰임

|:---|:---|:---|

소켓 프로그래밍: 서버와 클라이언트 구분<br>
클라이언트가 연결 요청 connect() → 서버가 accept() → 송수신 열림 → <br>
서버: 1.소켓 생성 socket() → 2.소켓과 종단점 주소 결합 bind() → 3.클라이언트 연결 대기 listen() → 4.클라이언트 연결 수용 accept() → 5.데이터 수신 recv() → 6.데이터 송신 send()<br>
클라이언트: 1.소켓생성 socket() → 2. 연결 요청 connect() → 3. 데이터 송신 send() → 4. 데이터 수신 recv()<br>
서버가 먼저 실행되고 listen() 상태까지가야 클라이언트가 connect 했을때 응답 할 수 있음

RS-232C: 단방향 또는 양방향 RS-485: 양방향

UART(Universal Asynchronous Receiver/Transmitter, 범용 비동기 송수신기)
• 두 장치 사이에서 직렬 데이터를 교환할 때 적용되는 프로토콜(규칙 세트)
• UART는 매우 간단하며 양방향으로 데이터를 송신 및 수신하기 위해 송신기와 수신기 사이에 두 개의 와이어만 사용
• 와이어 양 끝단은 접지 연결이 되어 있습니다. UART를 이용한 통신은 Simplex(단방향 통신)(데이터가 한 방향으로만 전송됨), Half-duplex(반이중)(한 번에 한 쪽만 전송 가능) 또는 Full-duplex(전이중)(양쪽이 동시에 전송 가능) 방식이있음
• UART에서 데이터는 프레임 형태로 전송

이더넷 케이블마다 속도가 다르고, 숫자가 높아질수록 빠름 100Mbps ~ 10Gbps

# 02장: 리눅스와 운영체제
운영체제: 사용자가 컴퓨터 사용하기 위해 필요한 소프트웨어, 컴퓨터 사용하면서 실행한 모든 프로그램들은 운영체제에서 관리하고 제어<br>
목적: 하드웨어 관리+사용자 편의 제공; 컴퓨터 성능 높이고 사용자에게 편의성 제공을 목적으로 함<br>
리눅스: 유닉스 기반해 만든 운영체제

운영체제 구성요소: 커널; 파일 시스템; 디바이스 드라이버; 시스템 호출; 인터페이스<br>
운영체제 기능: 프로세스 관리; 메모리 관리; 파일 시스템 관리; 입출력 관리; 자원 관리 및 보호; 사용자 인터페이스 제공<br>
커널: 하드웨어 초기화해 사용할 수 있게하는 운영체제 핵심 부분<br>
파일, 파일 시스템: 파일: 정보 저장하는 기본 단위; 파일 시스템: HDD, SSD등 저장 장치에 파일 저장 후 관리하는 소프트웨어

리눅스의 특징:리눅스는 공개 소프트웨어이며 무료로 사용, 유닉스와 완벽한 호환성을 유지, 서버용 운영체제로 많이 사용, 편리한 GUI 환경을 제공<br>
리눅스 종류: SLS(최초 배포판) → 슬랙웨어(살아있는 가장 오래된 배포판) → 수세(유럽에서 인기) → 레드햇(서버용) → 페도라(레드햇 계열, 레드햇은 무료배포 종료했는데 페도라가 유지)<br>

# 03장: ROS2 입문
센서: 측정대상으로부터 정보를 검지 혹은 측정해 측정량을 인식가능한 유용한 신호로 변환시키는 장치<br>
신호체계: 입력(센서) → 프로세싱(하드웨어 소프트웨어 펌웨어 OS 등) → 출력(Actuator)<br>
PWM(Pulse Width Modulation, 펄스 폭 변조): 디지털 신호 중 특정한 형태를 띈 신호를 일컫는 용어. 이 PWM은 회로제어에 다양한 용도로 활용. 가장 손쉽게 PWM을 사용한 예는 RGB LED의 색상변경, 서보 모터의 방향 전환에서 목격할 수 있음. 양 쪽 예 모두 디지털 신호를 PWM을 사용해 아날로그 신호와 유사한 효과를 냄. 이 방법은 일정한 주기의 디지털 신호의 출력이 HIGH인 시간과 LOW인 시간의 비율을 조정해서 아날로그 효과를 내는 방법, 디지털 신호만을 사용하는 마이크로 컨트롤러에서 아날로그 신호를 만들 필요가 있을 때 매우 유용하게 사용.<br>
아두이노 출력 5V=ON, 0V=OFF, 시간단위 ms(밀리초)

센서 → A/D변환(아날로그 신호 → 샘플링 → 양자화 → 부호화) → 디지털 정보 → 인식 → 구동기<br>
센서 선택시 고려해야하는 것: 가격, 크기 및 무게, 출력 형태, 인터페이스, 분해능, 감도, 선형성, 범위, 응답시간, 신뢰성, 정확성, 반복 정밀도<br>
거리 센서 종류: 카메라 센서, 라이다 센서, 초음파 센서, 적외선 센서, 레이다 센서

### 펌웨어와 미들웨어
||Firmware|Middleware|
|:---|:---|:---|
|위치|하드웨어와 가까운 저수준 소프트웨어<br>ROM이나 플래시 메모리와 같은 비휘발성 메모리에 저장|소프트웨어와 소프트웨어<br>하드웨어와 소프트웨어 사이의 중간 계층|
|역할|하드웨어 제어 및 초기화|시스템 간의 데이터 통신과 상호작용 관리|
|사용 예|마이크로컨트롤러, 임베디드 시스템|네트워크 통신, 데이터베이스접근, 여러 소프트웨어 모듈간의 연결
|변경 용이성|하드웨어에 내장되어 있어 수정이나 업데이트가 어려움|보통 업데이트가 가능하며, 시스템간의 통신에만 영향을 미침|
|사용자와의 관계|사용자는 거의 직접적으로 펌웨어를 다루지 않음|사용자 및 애플리케이션 미들웨어를 통해 시스템을 다룸|
|특징|하드웨어와의 직접적인 상호작용|여러 시스템간의 통신을 처리하는 소프트웨어|
|예시|마이크로 컨트롤러에서 실행되는 소프트웨어<br>라우터의 부팅과정에서 작동하는 소프트웨어<br>프린터의 하드웨어 동작을 제어하는 소프트웨어|데이터베이스 미들웨어<br>메시징 미들웨어<br>ROS(다양한 하드웨어 및 소프트웨어 모듈간의 통신을 중개)

### SBC vs 제어기
|          SBC||         제어기
-더 복잡하고 다양한 작업(예: 동영상 처리, 네트워크 서버 운영, 프로그래밍 학습)을 수행
-일반적인 컴퓨터처럼 다목적 사용이 가능
-운영 체제와 소프트웨어를 실행할 수 있어 복잡한 데이터 처리와 멀티태스킹에 적합
-특정한 작업(예: 온도 제어, 로봇 팔 움직임 제어, 센서 데이터 수집 및 처리)을 효율적으로 수행
-실시간 작업과 빠른 반응이 중요한 경우에 적합
-에너지 소모가 적고, 비용이 저렴하며, 단순한 시스템에 최적화
-보통 Linux, Windows IoT, Android와 같은 범용 운영 체제를 실행할 수 있음
-복잡한 소프트웨어와 GUI 기반의 애플리케이션 실행 가능
-대부분 운영 체제가 없거나, RTOS(Real-Time Operating System)와 같은 경량 운영 체제 사용
-작업이 운영 체제에 의존하지 않고, 펌웨어 수준에서 직접 작동
- CPU, GPU, RAM 등의 사양이 훨씬 높음
- USB, HDMI, 이더넷 포트 등 다양한 연결 옵션 제공
-복잡한 연산과 데이터 처리가 가능하며, 고해상도 디스플레이 출력도 지원

# 04장: ROS 소개
ROS: 오픈소스, 모듈 구조, 확장성<br>
ROS=plumbing + tools + capabilities + community

ROS 리눅스 사용하는 이유: 운영체제 라이센스 문제, 개발 시간 단축, 편리한 디버깅 도구 및 시각화 도구, 많은 사용자 및 생태계<br>
장점: 풍부한 생태게, 모듈화 및 재사용성, 다양한 하드웨어 지원<br>
단점: 학습곡선, 성능, 리소스 사용량

|ROS1|ROS2|
|:---|:---|
|단일 로봇|복수대의 로봇|

ROS1과 ROS2 비교
1. Platforms: ROS1은 리눅스만, ROS2는 윈도우 맥 리눅스 전부.
2. Real-time: ROS 2는 Real-time을 지원, 단, 선별된 하드웨어 사용, 리얼타임 운영체제 사용, DDS의
RTPS(Real-time Publish Subscribe Protocol) 와 같은 통신 프로토콜을 사용, 매우 잘 짜여진 리얼타임 코드 사용을 전제로 실시간성을 지원.
1. Security: ROS2는 DDS-Security 도입
2. Communication: TCPROS 자체 라이브러리 사용 → DDS 사용
3. Middleware interface: ROS 미들웨어 지원
4. Node Manager(discovery): roscore → 삭제(DDS로 Dynamic Discovery 사용)
5. Languages: ROS1: C++03, Python2.7; ROS2: C++17, Python 3.8
6. Build system: ROS1: catkin(CMake만 지원); ROS2:ament(catkin 업그레이드, Python 패키지 관리 가능)
7. Build tools: ROS1:catkin_make, "_"_isolated 및 "_tools 지원; ROS2: ament tools 도입 colcon build
8.  Build options: Multiple Workspace, No non-isolated Build, No devel space
9.  Version control system: ROS1: rosws → wstool → vcstool ROS2꺼 가져옴; ROS2: vcstool로 통합
10. Client library: RCL(ROS cli~ lib~), rclpy
11. Life Cycle: ROS1:SMACH 같은 독립 패키지 사용; ROS2: lifecycle을 클라이언트 라이브러리에 추가
12. Multiple nodes: ROS1: 초기에 하나의 프로세스에서 여러 노드 실행 불가; ROS2: components 라이브러리가 RCL에 포함됨. Intra process communication 기능도 추가해 통신 오버헤드 제거 가능
13. Threading model: ROS1: 단일 혹은 다중 스레드 실행 중 하나만 선택 가능; ROS2: 세분화된 단일 다중 스레드 RCL API로 쉽게 구현 가능
14. Messages(topic, service, action): ROS1: 단일 데이터 구조를 메시지라고 정의; ROS2: IDL(Interface Description Language)을 사용, 메시지 정의 및 직렬화를 더 쉽게, 더 포괄적으로 다룸 
15. Command Line Interface: 대부분 CLI 타입 명령어 사용법 기존 ROS 1과 매우 비슷해 약간의 이름 변경과 일부 옵션 사용법만 익히면 사용시 큰 차이는 없음.
16. roslaunch: run: 단일 프로그램 실행, launch: 사용자 지정 프로그램 실행; launch가 사용빈도 높음; ROS1: 'XML'로 실행; ROS2:'XML', 'YAML', 'Python' 다 사용 가능
17. Graph API: rqt_graph 제공
18. Embedded System: RTOS, DDS-XRCE, 블루투스 및 와이파이 통신까지 지원

DDS(Data Distribution Service): 데이터 통신을 위한 미들웨어, IDL 사용해 메시지 정의 및 직렬화, DDSI-RTPS 채용해 실시간 데이터 전송 보장 및 임베디드 시스템에 적용, QoS 설정 가능해 데이터 손실 방지 우선이나 실시간 통신 최우선 중시 가능.  DDS의 채용은 ROS 1에서 ROS 2로 바뀌면서 가장 큰 변화점, 개발자 및 사용자로 하여금 통신 미들웨어에 대한 개발 및 이용 부담을 줄여 진짜로 집중해야 할 부분에 더 많은 시간을 쏟을 수 있게 함.

### DDS 특징:
1. ​Industry Standards: 비영리단체가 관리하고 있어 산업 표준임
2. OS Independent: 다양한 운영체제를 지원해 사용하던 운영체제를 변경할 필요가 없음
3. Language Independent: DDS는 미들웨어이기에 그 상위 레벨이라고 볼 수 있는 사용자 코드 레벨에서는 DDS 사용을 위해 기존에 사용하던 프로그래밍 언어를 바꿀 필요가 없음
4. Transport on UDP/IP: UDP 기반이라 여러 목적지에 동시 데이터 전송 가능
5. Data Centricity: 발간 및 구독 방식, DDS의 미들웨어를 사용자 입장에서 본다면 어떤 데이터인지, 이 데이터가 어떤 형식인지, 이 데이터를 어떻게 보낼 것인지, 이 데이터를 어떻게 안전하게 보낼 것인지에 대한 기능이 DDS 미들웨어에 녹여있기 때문
6. Dynamic Discovery: 응용 프로그램은 DDS의 동적 검색을 통해 어떤 토픽이 지정 도메인 영역에 있으며 어떤 노드가 이를 발신하고 수신하는지 알 수 있게 된다
7. Scalable Architecture:  Participant 형태의 노드는 확장 가능한 형태로 제공되어 사용가능, 단일 표준 통신 계층에서 많은 복잡성을 흡수해 분산 시스템 개발을 더욱 단순화 시켜 편의성을 높임. 
8. Interoperability: 상호 운용성 지원, 10여개 회사가 DDS 통신 지원, 그 중 5개가 ROS2 지원(구름, 이프록시마, 이클립스)
9. Quality of Service (QoS): 데이터 손실 방지 우선이나 실시간 통신 최우선 중시 가능
10. Security:  DDS-Security이라는 DDS 보안 사양을 ROS에 적용하여 보안에 대한 이슈를 통신단부터 해결하였다. 또한 ROS 커뮤니티에서는 SROS2(Secure Robot Operating System 2) 툴 개발해 배포

ROS 통신: 노드, 토픽, 서비스, 액션
노드: 최소 단위의 실행 가능한 프로세스, 하나의 실행 가능한 프로그램
패키지: 하나 이상의 노드 또는 노드 실행을 위한 정보 등을 묶어 놓은 것, 패키지의 묶음을 메타패키지(metapackage)라 하여 따로 분리
수많은 노드들이 연동되는 ROS 시스템을 위해서는 노드와 노드 사이에 입력과 출력 데이터를 서로 주고받게 설계:
메시지: 노드들이 주고받는 데이터, integer, floating point, boolean, string 와 같은 변수 형태, 메시지 안에 메시지를 품고 있는 간단한 데이터 구조 및 메시지들의 배열과 같은 구조도 사용가능
메시지 통신: 메시지를 주고받는 방식. 메시지를 주고받는 통신 방법에 따라 토픽(topic), 서비스(service), 액션(action), 파라미터(parameter)로 구분

ROS 구성요소
노드 
- ROS에서 실행되는 최소단위의 프로세스를 말함 
- ROS에서는 하나의 목적에 하나의 노드 작성을 권장함 
- 노드가 구동될 때, 노드의 역할(토리그 서비스 등), 메시지 형태 등이 등록됨
패키지
- ROS를 구성하는 기본 단위
- ROS 응용프로그램은 패키지 단위로 개발되며 최소한 하나의 노드를 가짐 
메시지
- 노드와 노드는 서로 메시지를 통해서 데이터를 주고받음 
- 메시지는 integer, float, Boolean 등의 변수 형태임 
- 메시지안에 메시지를 품는 데이터 구조가 가능
- 단방향은 토픽(topic), 양방향은 서비스(service) 
토픽
- 토픽은 ROS의 노드 사이의 데이터 통신의 한 종류 
- 단방향 통신으로, 데이터를 메시지에 담아서 방송을 함 
- 방송을 하는 퍼블리셔(publisher)와 수신을 하는 서브스크라이버(subscriber)로구성 됨 

토픽: 비동기식 단방향 메시지 송수신 방식, msg메시지 형태의 메시지를 발간하는 Publisher와 메시지를 구독하는 Subscriber 간의 통신. 이는 1:N, N:1, N:N 통신도 가능, ROS 메시지 통신에서 가장 널리 사용되는 통신 방법

서비스: 동기식 양방향 메시지 송수신 방식, 서비스의 요청(Request)을 하는 쪽은 Service client, 서비스의 응답(Response)을 하는 쪽은 Service server. 결국 서비스는 특정 요청을 하는 클라이언트 단과 요청받은 일을 수행 후에 결과 값을 전달하는 서버 단과의 통신. 서비스 요청 및 응답(Request/Response) 또한 msg 메시지의 변형으로 srv메시지라고 한다

액션: 비동기식+동기식 양방향 메시지 송수신 방식, 액션 목표 Goal를 지정하는 Action client과 액션 목표를 받아 특정 태스크를 수행하면서 중간 결과 값에 해당되는 액션 피드백(Feedback)과 최종 결과 값에 해당되는 액션 결과(Result)를 전송하는 Action server 간의 통신.<br>
파라미터: 각 노드에 파라미터 관련 Parameter server를 실행시켜 외부의 Parameter client 간의 통신으로 파라미터를 변경하는 것으로 서비스와 동일하다고 볼 수 있다. 단 노드 내 매개변수 또는 글로벌 매개변수를 서비스 메시지 통신 방법을 사용하여 노드 내부 또는 외부에서 쉽게 지정 하거나 변경할 수 있고, 쉽게 가져와 사용할 수 있게 하는 점에서 목적이 다르다.

ros2 run turtlesim turtle_teleop_key --ros-args --remap turtle1/cmdvel:=turtle33/cmd_vel<br>
--ros-args --remap: arguments(--remap은 arguments 이름, 여러개 있음)<br>

### ROS2 실습 - Topic
• `Node A`처럼 하나의 이상의 토픽을 발행
• `Publisher` 기능과 동시에 토픽(예: Topic D)을 구독하는 `Subscriber` 역할도 동시에 수행
• 자신이 발행한 토픽을 셀프 구독할 수 있게 구성할 수도 있음
• 토픽 기능은 목적에 따라 다양한 방법으로 사용할 수 있으며 이런 유연성으로 다양한 곳에 사용 중
• ROS 프로그래밍시에 70% 이상이 토픽으로 사용됨
• 통신 방식 중에 가장 기본이 되며 가장 널리쓰이는 방법
• 비동기성과 연속성을 가지기에 센서 값 전송 및 항시 정보를 주고 받아야 하는 부분에 주로 사용

### ROS2 실습 - Service
• 서비스는 다음 그림과 같이 동일 서비스에 대해 복수의 클라이언트를 가질 수 있도록 설계되었다.
• 단, 서비스 응답은 서비스 요청이 있었던 서비스 클라이언트에 대해서만 응답을 하는 형태
• Node C의 Service Client가 Node B의 Service Server에게 서비스 요청을 하였다면 Node B의 Service Server는 요청 받은 서비스를 수행한 후 Node C의 Service Client에게만 서비스 응답

### ROS2 실습 - Action
액션(Action)
• 토픽(topic)과 서비스(service)의 혼합
• 액션 목표 및 액션 결과를 전달하는 방식은 서비스와 같고
• 액션 피드백은 토픽과 같은 메시지 전송 방식
• 액션 목표/피드백/결과(Goal/Feedback/Result) 메시지 또한 위에서 언급한 msg 메시지의 변형으로  action 메시지라고 함

Action Client는 Service Client 3개와 Topic Subscriber 2개로 구성되어있으며, Action Server는 Service Server 3개와 Topic Publisher 2개로 구성<br>
액션 목표/피드백/결과(goal/feedback/result) 데이터는 msg 및 srv 인터페이스의 변형으로 action 인터페이스라고 함.

ROS 1에서의 액션은 목표, 피드백, 결과 값을 토픽으로만 주고 받았는데 ROS 2에서는 토픽과 서비스 방식을 혼합하여 사용하였다. 그 이유로 토픽으로만 액션을 구성하였을 때 토픽의 특징인 비동기식 방식을 사용하게 되어 ROS 2 액션에서 새롭게 선보이는 목표 전달(send_goal), 목표 취소(cancel_goal), 결과 받기 (get_result)를 동기식인 서비스를 사용하기 위해서이다. 이런 비동기 방식을 이용하다보면 원하는 타이밍에 적절한 액션을 수행하기 어려운데 이를 원활히 구현하기 위하여 목표 상태(goal_state)라는 것이 ROS 2에서 새롭게 선보였다. 목표 상태는 목표 값을 전달 한 후의 상태 머신을 구동하여 액션의 프로스세를 쫒는 것이다. 여기서 말하는 상태머신은 Goal State Machine으로 다음 그림과 같이 액션 목표 전달 이후의 액션의 상태 값을 액션 클라이언트에게 전달할 수 있어서 비동기, 동기 방식이 혼재된 액션의 처리를 원활하게 할 수 있게 되어 있다.

ROS2 실습 - Interface<br>
ROS의 노드 간에 데이터를 주고받을 때에는 토픽, 서비스, 액션이 사용되는데 이 때 사용되는 데이터의 형태를 ROS 인터페이스(interface) 이라고 한다. ROS 인터페이스에는 ROS 2에 새롭게 추가된 IDL(interface definition language)과 ROS 1부터 ROS 2까지 널리 사용 중인 msg, srv, action이 있다. 토픽, 서비스, 액션은 각각 msg, srv, action interface를 사용하고 있으며 정수, 부동 소수점, 불리언과 같은 단순 자료형을 기본으로 하여 메시지 안에 메시지를 품고 있는 간단한 데이터 구조 및 메시지들이 나열된 배열과 같은 구조도 사용할 수 있다.

토픽은 고유의 인터페이스를 가지고 있는데 이를 메시지 인터페이스라 부르며, 파일로는 msg파일을 가르킨다. 예를 들어, 위 예제에서 /turtle1/cmd_vel 토픽은 geometry_msgs/msgs/Twist 형태이다. 이름이 좀 긴데 풀어서 설명하면 기하학 관련 메시지를 모아둔 geometry_msgs 패키지의 msgs 분류의 Twist 데이터 형태라는 것이다.

ROS2 실습 - Parameter<br>
파라미터 관련 기능은 RCL(ROS Client Libraries)의 기본 기능으로 다음 그림과 같이 모든 노드가 자신만의 Parameter server를 가지고 있고, 그림과 같이 각 노드는 Parameter client도 포함시킬 수 있어서 자기 자신의 파라미터 및 다른 노드의 파라미터를 읽고 쓸수 있게 된다. 이를 활용하면 각 노드의 다양한 매개변수를 글로벌 매개변수처럼 사용할 수 있게되어 추가 프로그래밍이나 컴파일 없이 능동적으로 변화 가능한 프로세스를 만들 수 있게 된다. 그리고 각 파라미터는 yaml 파일 형태의 파라미터 설정 파일을 만들어 초기 파라미터 값 설정 및 노드 실행시에 파라미터 설정 파일을 불러와서 
사용할 수 있기에 ROS 2 프로그래밍에 매우 유용하게 사용할 수 있다.
